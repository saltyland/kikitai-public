# チーム開発ガイド

このドキュメントは、開発チームにおける開発の進め方・ルールをまとめたものです。
GitHub の詳しい操作手順は [GITHUB_GUIDE.md](./GITHUB_GUIDE.md)を参照してください。

- GitHub リポジトリ: https://github.com/TakatoGokyu/softwareA-prac
- Notion チーム開発ハブ: 「ソフトウェア工学A チーム開発ハブ」（タスク管理 / ナレッジ管理 / 作業ログ / ロードマップ）

---

## 1. 開発の基本フロー

毎回の作業は以下の流れで行う。

```text
Notion でタスクのステータスを「進行中」にする
  ↓
main を最新化する (git pull origin main)
  ↓
作業用ブランチを作成する
  ↓
作業する
  ↓
commit する
  ↓
push する
  ↓
Pull Request を作成する → Notion タスクを「レビュー中」にする
  ↓
班員にレビューしてもらう
  ↓
Approve 後、main に merge する → Notion タスクを「完了」にする
  ↓
作業ログを Notion に記録する
```

### 基本ルール

- `main` ブランチに直接 push しない
- 作業前に必ず `git pull origin main` で最新化する
- 1つのブランチでは、なるべく1つの機能・タスクだけを作業する
- わからないエラーが出たら、すぐに班員に共有する
- 他の人が作業しているファイルを大きく変更する場合は、事前に相談する

具体的な git コマンドは [GITHUB_GUIDE.md](./GITHUB_GUIDE.md) を参照。

---

## 2. ブランチ運用ルール

ブランチ名は「何の作業か」が分かる名前にする。

| プレフィックス | 用途 | 例 |
|---|---|---|
| `feature/...` | 新機能の追加 | `feature/login`, `feature/start-stop` |
| `fix/...` | バグ修正 | `fix/server-error` |
| `docs/...` | ドキュメントの追加・修正 | `docs/readme` |

個人名を入れてもよい。

```text
feature/chiba-server
feature/kubota-client
```

---

## 3. コミットメッセージ規約

コミットメッセージは **日本語可**。先頭にプレフィックスを付ける。

| プレフィックス | 用途 |
|---|---|
| `feat:` | 新機能の追加 |
| `fix:` | バグ修正 |
| `docs:` | ドキュメントの変更 |
| `refactor:` | リファクタリング（挙動を変えない変更） |
| `style:` | フォーマット・見た目のみの修正 |
| `chore:` | その他（設定変更・依存関係の更新など） |

例:

```text
feat: ログイン機能を追加
fix: サーバー接続エラーを修正
docs: READMEを更新
```

## 4. Pull Request のルール

- PR のタイトルには「何をしたか」を簡潔に書く
- 説明欄には変更内容を箇条書きで書く（[GITHUB_GUIDE.md](./GITHUB_GUIDE.md) のテンプレート参照）
- **main への merge には、PR作成者以外の班員 1名以上のレビュー（Approve）を必須とする**
  - 理由: 誰もチェックしていないコードが main に入る事故を防ぐため
- レビューで指摘があれば修正し、再度確認してもらってから merge する
- Approve されたら `Merge pull request` で merge する
- merge 後、不要になったブランチは削除してよい

---

## 5. コンフリクトについて

- コンフリクトが発生したら、無理に進めず班員に確認する
- 解消後は、必要な内容だけを残して `<<<<<<<` `=======` `>>>>>>>` を削除し、再度 `git add` → `git commit` する

詳細は [GITHUB_GUIDE.md](./GITHUB_GUIDE.md) の「12. コンフリクトについて」を参照。

## 6. Notion 運用ルール

Notion「ソフトウェア工学A チーム開発ハブ」には以下の4つのデータベースがある。

| データベース | 用途 |
|---|---|
| タスク管理 | 各タスクの進捗・担当者・優先度・期限を管理 |
| ナレッジ管理 | 調査結果・参考サイト・論文・議事録・開発ルールなどの知見を蓄積 |
| 作業ログ | 各作業セッションでやったことを記録 |
| ロードマップ | フェーズごとの開発計画 |

### 6-1. タスク管理のステータス遷移

タスク管理DBの「ステータス」は、以下のタイミングで切り替える。

| ステータス | 切り替えタイミング |
|---|---|
| 未着手 → 進行中 | 作業に着手するとき |
| 進行中 → レビュー中 | Pull Request を作成したとき |
| レビュー中 → 完了 | Pull Request が main に merge されたとき |

### 6-2. 作業ログの記録

- **1作業セッションごと**に「やったこと」を作業ログDBに追加する
- 関連する「タスク管理」「ロードマップ」「ナレッジ管理」を可能な範囲で紐付ける

### 6-3. ナレッジ管理への蓄積

- リサーチした結果・参照したサイト・論文・議事録・開発ルールの変更などは、すべてナレッジ管理DBに追加する
- トピックタグ（要件定義 / 開発ルール / 調査 / 議事録 / 進行予定）を付けて分類する

---

## 7. 困ったとき

- エラーや疑問点は一人で抱え込まず、すぐに班員に共有する
- ルール自体への疑問・改善案があれば、ナレッジ管理DB（トピック: 開発ルール）に追記し、チームで議論する
