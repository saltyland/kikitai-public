## 0. これは何のプロジェクト？

**「キキタイ」**＝学生・研究者のための **P2P型の学術アンケート交換プラットフォーム**（Webアプリ）。

### 解こうとしている課題
研究や授業課題でアンケートを作っても「回答者が集まらない」。逆に、回答する側にはメリットがない。
そこで **“回答し合う”経済圏** を作る：

1. 他人のアンケートに**回答するとポイントが貯まる**。
2. 貯めたポイントを使って、**自分のアンケートに回答者を集める**（＝公開コストを払う想定）。
3. 回答の質を **AIが自動評価**し、雑な回答は報酬倍率0／信頼スコア減点。良質な回答は×1.5。
   → 「とりあえず埋めるだけ」の回答を抑え、データの質を担保する。
4. プロフィール属性は**非公開にするほどポイントボーナス**がもらえる逆インセンティブ設計
   （非公開にした属性はマッチング＝ターゲティングに使われない代わりにボーナス）。

### 主要な機能（実装済み）
- **アカウント**：Supabase Auth でメール＋パスワード登録／ログイン（メール確認はOFF運用）。
- **アンケート作成**：Googleフォーム相当。設問タイプ＝単一/複数選択・プルダウン・記述/段落・日付・
  スケール（可変段階＋両端ラベル）・グリッド。必須トグル・説明文・**セクション（ページ分割）**・
  **表示条件（分岐）**・ドラッグ&ドロップ並べ替え・複製・テンプレートライブラリ・左右2ペインの即時プレビュー。
- **回答**：1問ずつのステッパー形式。進捗表示・途中保存（localStorage）・送信前確認・スマホ最適化・
  オフライン時の再送（Service Worker）。
- **集計／結果**：作成者のみ閲覧。円グラフ＋棒グラフ、グリッドのクロス集計、CSVダウンロード。
  Proプラン限定の**統計解析モード**（平均・中央値・標準偏差・最頻値）。
- **AI品質評価＋ポイント**：上記のとおり（直前にやったこと §2/§3）。
- **料金プラン**：free / pro（デモは決済なし、トグルで切替）。

## 3. アーキテクチャ（OOP）

UIから直接DBアクセスしない。層を分離：

```
app/                ページ＋ app/actions/（サーバーアクション）
components/          UI（クライアント/サーバー）。ui/ は共通デザイン部品（Button/Card）
lib/
  supabase/         client(browser) / server / admin(service_role) / proxy
  services/         ビジネスロジック（AuthService / ResponseService / ProfileService 等）
  repositories/     DBアクセス（BaseRepository<T> を継承）
  domain/questions/ 設問タイプの多態（後述）
  domain/quality/   AI品質評価（IQualityEvaluator / Gemini / ルールベース）
  utils.ts          cn（クラス名結合）
  types/
proxy.ts            認証ガード
supabase/migrations/  DBスキーマ（冪等マイグレーション）
scripts/supabase-sync.mjs  npm run sync の実体
```

- **設問タイプの多態**：`QuestionTypeDefinition`（抽象親）→ Choice系 / Text / Grid 等。`QuestionTypeRegistry`
  経由でのみ利用。**新タイプはクラス追加＋レジストリ登録だけ**で済む設計（`pointCost` も各タイプが保持）。
- **品質評価の多態（Strategy）**：`IQualityEvaluator` を `createQualityEvaluator()` で切替。
- **リポジトリ抽象基底**：`BaseRepository<T>`（共通CRUD）。

## 4. Supabase 運用

- スキーマは `supabase/migrations/` でバージョン管理（**冪等**＝再実行安全。`add column if not exists` 等）。
- **手動でSQLをダッシュボードに貼らない**。`npm run sync` か `npm run db:push` で反映。
- `npm run sync`（`scripts/supabase-sync.mjs`）：
  1. Management API で **メール確認OFF（mailer_autoconfirm=true）** を設定
  2. マイグレーション反映：**DBパスワードがあれば `supabase db push`**、無ければ Management API の
     query エンドポイントでSQLを直接実行（`SUPABASE_ACCESS_TOKEN` だけで反映可能）。
- 必要な秘密情報（`.env.local`・git管理外）：
  - `SUPABASE_ACCESS_TOKEN`（https://supabase.com/dashboard/account/tokens）
  - `SUPABASE_DB_PASSWORD`（db push に必要。未設定だとpushはスキップ）
  - `SUPABASE_SERVICE_ROLE_KEY`（退会でauth.usersごと削除する場合）
  - `GEMINI_API_KEY`（任意。あれば AI評価が Gemini 動作。無ければルールベース）
- プロジェクト参照（ref）：`cpyhkdlbwsiiwwucpkji`
- メール確認はOFF済み（登録すると即ログイン）。OFF前の未確認アカウントは Authentication → Users から削除。
