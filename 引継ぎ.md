## ＜以前＞ ポイント消費を「公開時一括」→「回答ごとの品質比例」へ変更

マイグレ `20260614110000_pay_per_answer.sql` を追加（**`npm run sync` 反映済み**）。

- **公開時はポイントを消費しない**：`publish_survey` v2 は open 化のみ。無一文での回答集め
  防止のため「最低残高＝1回答分（設問コスト合計、最低1pt）」だけチェック
  （不足は INSUFFICIENT_POINTS。dbError のメッセージも「公開には最低◯pt（1回答分）」に変更）。
- **回答ごとの品質比例課金**：`submit_survey_response` v3 が、回答者への付与額
  （品質倍率 0／×1.0／×1.5 × 設問コスト合計）と**同額を作成者から消費**する。
  低品質回答ほど作成者のコストは安く、高品質ほど高い。消費は新RPC
  `consume_points_upto`（FIFO・**残高の範囲内のみ・不足でもエラーにしない**）で行い、
  作成者が残高不足でも回答送信は成功・回答者は満額受け取る（回答者を巻き込まない）。
  **author_refund（高品質時の作成者還元）は本モデルと矛盾するため廃止**
  （旧6引数シグネチャは drop、`ResponseService`/`ResponseRepository` から引数削除）。
- **作成画面にコスト目安**：`SurveyEditor` の操作ボタン上に「1回答あたり 平均◯pt
  （品質により0〜◯pt）／必要回答数◯件で合計約◯pt／公開には最低◯pt」パネルを表示
  （TSの `pointCost` から算出。DB側 `question_point_cost` と要同期）。
- **回答一覧に報酬目安**：`listAnswerableSurveys` が新リポジトリメソッド
  `findQuestionTypesBySurveyIds`（1クエリ）で設問タイプを取得し
  `SurveyWithStats.avg_reward_points / max_reward_points` を付与。`SurveyCard` の
  フッターに「全問回答で平均◯pt（最高◯pt）」チップを表示。
- `PointsSummaryCard` の注記文言を都度課金の説明に変更。

検証：`tsc --noEmit` / `npm run lint` / `npx next build` / `npm run sync` 通過。
ブラウザでの画面確認は未実施（ログイン必要のため）。

## ＜以前＞ ポイント経済の整合性＋プロフィールRLS是正

マイグレ `20260612120000_point_economy.sql` を追加（**`npm run sync` 反映済み**。RPC・ビュー・
ポリシーの存在をManagement APIで確認済み）。ビジネスエラーは `KIKITAI:<CODE>:<詳細>` 形式の
`raise exception` → `lib/repositories/dbError.ts` の `throwRpcError` が `BusinessRuleError`
（日本語メッセージ）へ変換する。

- **公開時ポイント消費**：`publish_survey` RPC（security definer・1トランザクション）が
  コスト＝`ceil(required_count × 設問単価合計)` を計算し、`consume_points`（granted_at昇順FIFO・
  `for update` 行ロック・束の分割/削除）で消費して open 化。残高不足は INSUFFICIENT_POINTS で
  公開拒否。`SurveyService.changeStatus` の draft→open がこれを呼ぶ。失敗時はホームに赤バナー
  （`/?statusError=`）。設問単価のSQL版 `question_point_cost` は **TSの `pointCost` と要同期**。
- **回答送信のアトミック化**：`submit_survey_response` RPC が回答保存＋報酬付与＋信頼スコア減点＋
  **required_count到達時の自動close** を1トランザクションで実行（earned_pointsはDB側でも
  コスト×1.5に上限クランプ）。`ResponseService.submitResponse` はAI評価後にこれを1回呼ぶだけ。
  締切時はホームに「締め切られました」表示（`closed=1`）。
- **残高キャッシュ**：`sync_points_balance` RPC（1文のUPDATE内で集計）で lost update を解消。
  `ProfileService.syncPointsBalance` はRPC委譲に変更。
- **プロフィールRLS**：`profiles` の `select using(true)` を廃止し本人のみに。他人の情報は
  `public_profiles` ビュー（security definer・`private_fields` の属性を null マスク、
  points/trust_score/plan は非公開）経由のみ。`ProfileRepository.findByIds` はビュー参照に変更
  （戻り値は新型 `PublicProfile`）。
- **UI**：`SurveyCard` に回答進捗バー（response_count/required_count）を追加。
- **UI（2026-06-12 追記）**：一覧 `/surveys` の設問プレビューの選択肢をピル横並び→縦並びの
  ラジオ/チェック風リストに変更（`SurveyCard.tsx` の `MiniInput`）。カード間の縦間隔を
  `gap-y-8`→`gap-y-14` に拡大（カード下の投稿者アバターが上下どちらのカードのものか
  分かりづらかったため）。
- **整理**：未使用化した `saveResponse`/`awardPoints`/`adjustTrust`/`setPoints`/`setTrustScore`/
  `isUniqueViolation` を削除（処理はRPCへ集約）。

検証：`tsc --noEmit` / `npm run lint` / `npx next build` / `npm run sync` 通過。

## 0. これは何のプロジェクト？

**「キキタイ」**＝学生・研究者のための **P2P型の学術アンケート交換プラットフォーム**（Webアプリ）。

### 解こうとしている課題
研究や授業課題でアンケートを作っても「回答者が集まらない」。逆に、回答する側にはメリットがない。
そこで **“回答し合う”経済圏** を作る：

1. 他人のアンケートに**回答するとポイントが貯まる**。
2. 貯めたポイントを使って、**自分のアンケートに回答者を集める**（＝公開コストを払う想定）。
3. 回答の質を **AIが自動評価**し、雑な回答は報酬倍率0／信頼スコア減点。良質な回答は×1.5。
   → 「とりあえず埋めるだけ」の回答を抑え、データの質を担保する。
4. プロフィール属性は**非公開にするほどポイントボーナス**がもらえる逆インセンティブ設計
   （非公開にした属性はマッチング＝ターゲティングに使われない代わりにボーナス）。

### 主要な機能（実装済み）
- **アカウント**：Supabase Auth でメール＋パスワード登録／ログイン（メール確認はOFF運用）。
- **アンケート作成**：Googleフォーム相当。設問タイプ＝単一/複数選択・プルダウン・記述/段落・日付・
  スケール（可変段階＋両端ラベル）・グリッド。必須トグル・説明文・**セクション（ページ分割）**・
  **表示条件（分岐）**・ドラッグ&ドロップ並べ替え・複製・テンプレートライブラリ・左右2ペインの即時プレビュー。
- **回答**：1問ずつのステッパー形式。進捗表示・途中保存（localStorage）・送信前確認・スマホ最適化・
  オフライン時の再送（Service Worker）。
- **集計／結果**：作成者のみ閲覧。円グラフ＋棒グラフ、グリッドのクロス集計、CSVダウンロード。
  Proプラン限定の**統計解析モード**（平均・中央値・標準偏差・最頻値）。
- **AI品質評価＋ポイント**：上記のとおり（直前にやったこと §2/§3）。
- **料金プラン**：free / pro（デモは決済なし、トグルで切替）。

## 3. アーキテクチャ（OOP）

UIから直接DBアクセスしない。層を分離：

```
app/                ページ＋ app/actions/（サーバーアクション）
components/          UI（クライアント/サーバー）。ui/ は共通デザイン部品（Button/Card）
lib/
  supabase/         client(browser) / server / admin(service_role) / proxy
  services/         ビジネスロジック（AuthService / ResponseService / ProfileService 等）
  repositories/     DBアクセス（BaseRepository<T> を継承）
  domain/questions/ 設問タイプの多態（後述）
  domain/quality/   AI品質評価（IQualityEvaluator / Gemini / ルールベース）
  utils.ts          cn（クラス名結合）
  types/
proxy.ts            認証ガード
supabase/migrations/  DBスキーマ（冪等マイグレーション）
scripts/supabase-sync.mjs  npm run sync の実体
```

- **設問タイプの多態**：`QuestionTypeDefinition`（抽象親）→ Choice系 / Text / Grid 等。`QuestionTypeRegistry`
  経由でのみ利用。**新タイプはクラス追加＋レジストリ登録だけ**で済む設計（`pointCost` も各タイプが保持）。
- **品質評価の多態（Strategy）**：`IQualityEvaluator` を `createQualityEvaluator()` で切替。
- **リポジトリ抽象基底**：`BaseRepository<T>`（共通CRUD）。

## 4. Supabase 運用

- スキーマは `supabase/migrations/` でバージョン管理（**冪等**＝再実行安全。`add column if not exists` 等）。
- **手動でSQLをダッシュボードに貼らない**。`npm run sync` か `npm run db:push` で反映。
- `npm run sync`（`scripts/supabase-sync.mjs`）：
  1. Management API で **メール確認OFF（mailer_autoconfirm=true）** を設定
  2. マイグレーション反映：**DBパスワードがあれば `supabase db push`**、無ければ Management API の
     query エンドポイントでSQLを直接実行（`SUPABASE_ACCESS_TOKEN` だけで反映可能）。
- 必要な秘密情報（`.env.local`・git管理外）：
  - `SUPABASE_ACCESS_TOKEN`（https://supabase.com/dashboard/account/tokens）
  - `SUPABASE_DB_PASSWORD`（db push に必要。未設定だとpushはスキップ）
  - `SUPABASE_SERVICE_ROLE_KEY`（退会でauth.usersごと削除する場合）
  - `GEMINI_API_KEY`（任意。あれば AI評価が Gemini 動作。無ければルールベース）
- プロジェクト参照（ref）：`cpyhkdlbwsiiwwucpkji`
- メール確認はOFF済み（登録すると即ログイン）。OFF前の未確認アカウントは Authentication → Users から削除。
